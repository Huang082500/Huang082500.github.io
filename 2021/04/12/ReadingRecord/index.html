《Unity 游戏优化》阅读笔记

2021-04-12 约 764 字 预计阅读 2 分钟 次阅读

文章目录

  • 优化核心思想
  • 脚本优化

优化核心思想

1.过早的优化是万恶之源。（在可验证的情况下进行优化和重构）

2.逻辑优化的最简洁列表：

  1. 验证目标脚本是否出现在场景中。
  2. 验证脚本在场景中出现的次数是否正确。
  3. 验证脚本的正确顺序。
  4. 最小化正在进行的代码更改。
  5. 尽量减少内部干扰。
  6. 尽量减少外部干扰。

脚本优化

  1. GetComponent() 比 GetComponent(string name) 性能优越。避免产品中使用 GetComponent(string name)。

  2. 使用 .CompareTag() 替换 .tag == “XX” 方法。

  3. 使用 Instantiate(gameobject,transform) 代替 Instantiate(gameobject).SetParent(transform). 因为Unity 5.4 以后 Transform 的层次结构改成了动态数组，初始化 GameObject 后再改变它的父节点，会使得当前父节点的 Transform 对应的缓存动态数组发生改变。

  4. 通过使用距离平方而不是距离禁用对象：方法为 OnBecameVisible() 和 OnBecameInvisible(), GameOject 还必须附加一个可渲染的组件，例如 MeshRenderer 或者 SKinnedMeshRenderer 来触发可见性回调，因为该回调必须与渲染管道通信。

  5. 协程会在包含他的 GameObject 变成不活动的那一刻自动停止，如果 GameObject 再次被设置为活动的，协程不会自动重新启动。

  6. Update 跟当前平台的帧数有关，FixedUpdate 是真实时间，所以处理物理逻辑的时候要把代码放在 FixedUpdate 而不是 Update。

关于 Unity 中的 Mono

Mono 是一个允许使用通用编程语言编写的代码可以运行在很多不同硬件平台上的开源框架。本质上，它是 .NET 类库的开源重制。mono 仅仅是 Unity 底层的一部分。

关于托管代码和本地代码

  1. 本地代码：比如 c/c++, 经过编译后生成汇编码，直接在 CPU 上运行，但是由于不同的硬件架构和操作系统，会导致不同的 CPU 支持的指令可能不同，这两种语言的移植很麻烦， 但是执行效率高。

  2. 托管代码：比如 Java, 使用本身框架的虚拟机，来屏蔽底层硬件和操作系统的不同，代码先经过虚拟机编译成中间语言，然后虚拟机将这些中间语言转换成其所在平台可以执行的本地机器码。

  3. 外部库：DirectX,OpenGL 库，也包含项目中包含的很多自定义库和插件。

  4. 托管代码调用本地代码，和托管代码调用外部库均会导致内存上下文切换。

  5. 托管代码(unity C#)的 GC 由 Mono 来执行。所以 Unity 上的 GC 其实指的是 Mono 的 GC 过程。
